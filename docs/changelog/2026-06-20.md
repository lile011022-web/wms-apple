## Login Page Employee Registration And Shared Data Refresh

### What Changed

- Added public employee registration through `POST /auth/register`.
- The login page now lets employees switch between 登录 and 注册账号.
- Registered employees are created as active `OPERATOR` users and are automatically logged in after account creation.
- The `OPERATOR` role gets operational permissions for shared warehouse workflows, but not system settings, user management, or role management.
- Seed data now creates both `ADMIN` and `OPERATOR` roles.
- The frontend query client now refetches active page data on mount, browser focus, and network reconnect.
- The topbar refresh action now refreshes all active page data queries instead of only refetching the current user.

### Touched Modules

- `apps/api/src/modules/auth`: owns login-page registration, session creation, and registration API documentation.
- `apps/api/src/modules/users`: owns public operator account creation, password hashing, duplicate-email validation, role assignment, and registration audit logs.

- `apps/web/src/api/query-client.ts`: owns cross-page refetch defaults for shared operational data.
- `apps/web/src/styles/globals.css`: styles the login/register segmented control and success state.
- `docs/api/03-auth.md` and `docs/api/04-users-roles-permissions.md`: document registration and operator permissions.

### Usage Logic

1. On the login page, choose 注册账号.
2. Enter employee name, email, and a password of at least 8 characters.
3. Submit to create an `OPERATOR` account and enter the WMS immediately.
4. All employees read and write the same backend PostgreSQL data through the API.
5. When another employee changes customer, inbound, inventory, outbound, or exception data, use the topbar 刷新数据 button or refocus the browser tab to fetch the latest server state.

## Inbound Draft Duplicate Guard And Row Delete

### What Changed

- Added same-draft IMEI and Serial duplicate validation before confirming an inbound draft.
- Confirmation now returns a clear business error such as `本次入库单内 IMEI 重复` instead of letting a database unique constraint surface as a generic server error.
- Added a row-level delete button to the inbound scan draft table so operators can remove duplicate or mistaken preview rows from the page.
- Wired the existing `DELETE /api/v1/inbound/drafts/:id/items/:itemId` endpoint into the web API client.

### Touched Modules

- `apps/api/src/modules/inbound/inbound.service.ts`: checks active confirmable draft rows for duplicate IMEI or Serial values before entering the repository confirmation transaction.
- `apps/api/src/modules/inbound/tests/inbound.service.spec.ts`: covers same-draft duplicate IMEI rejection and ensures the confirmation repository transaction is not called.
- `apps/web/src/pages/inbound-scan/page.tsx`: renders the delete action in the current draft table, calls the delete API, refreshes the draft, and shows success or error feedback.
- `apps/web/src/api/workflow.ts`: exposes `inboundApi.removeItem`.
- `apps/web/src/styles/globals.css`: extends table action styling for danger actions.
- `docs/api/08-inbound-scan.md`, `docs/database/05-inbound-inventory-transaction.md`, and `apps/web/src/api/integration-plan.ts`: document the delete endpoint usage and confirmation duplicate guard.

### Usage Logic

1. While the inbound batch is still `DRAFT`, each non-confirmed row shows a 删除 action.
2. Clicking 删除 logically voids the row, refreshes the current draft, and removes it from the active preview table.
3. Clicking 确认入库 checks all pending matched rows for repeated IMEI or Serial values in the same draft.
4. If a duplicate exists, the confirmation is rejected with a Chinese business message so the operator can delete the duplicate row or correct the scan before confirming.

## Frontend Token Refresh For Long Sessions

### What Changed

- Fixed the import workflow error `Authentication token is invalid or expired.` when the page has been open longer than the access-token lifetime.
- The frontend API client now attempts `POST /auth/refresh` automatically when a protected request returns an authentication error.
- If refresh succeeds, the original request is retried once, so actions such as inbound CSV import continue without forcing the operator to log in again.
- If refresh also fails, the local tokens are cleared and the page shows `登录已过期，请重新登录。`.

### Touched Modules

- `apps/web/src/api/client.ts`: handles authentication failures, refreshes tokens, retries the original request, and clears invalid sessions.
- `apps/web/src/api/client.test.ts`: covers successful refresh/retry and failed-refresh token clearing.
- `apps/web/src/layouts/app-layout.tsx`: listens for auth-expired events and returns the app to the login state.
- `docs/api/16-frontend-api-integration.md`: documents the refresh-and-retry behavior.

### Usage Logic

1. Operators can keep the WMS page open while working.
2. If the access token expires, the next protected action automatically refreshes the token.
3. If the refresh token is also expired, the operator is prompted to log in again before retrying the action.
