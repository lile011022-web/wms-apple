## 重复 IMEI 阻断确认入库

### 变更内容

- 确认入库前新增库存级 IMEI/Serial 重复校验。
- 如果当前入库单里有 IMEI 或 Serial 已经存在于库存，`确认入库` 会直接失败，整张草稿保持打开。
- 操作员必须先在当前入库单中修正或删除重复明细，才能继续确认入库。
- 后端事务内也保留同样的重复兜底校验，避免多人同时操作时重复身份写入库存。
- 原有同一草稿内重复 IMEI/Serial 阻断逻辑保持不变；物流单号重复仍按异常明细处理。

### 修改位置

- 后端服务：`apps/api/src/modules/inbound/inbound.service.ts`
  - `confirmDraft` 在提交事务前检查当前草稿明细是否已存在库存级 IMEI/Serial。
- 后端仓储：`apps/api/src/modules/inbound/inbound.repository.ts`
  - 事务内再次检查库存级 IMEI/Serial，防止并发确认造成重复入库。
- 后端测试：`apps/api/src/modules/inbound/tests/inbound.service.spec.ts`
  - 覆盖库存已存在 IMEI 时不能确认入库，且不会调用确认事务。
- 产品/API/数据库/架构文档：
  - `docs/product/05-inbound-rules.md`
  - `docs/api/08-inbound-scan.md`
  - `docs/database/05-inbound-inventory-transaction.md`
  - `docs/architecture/backend-implementation-roadmap.md`

### 使用逻辑

1. 操作员正常扫码加入当前入库单。
2. 点击 `确认入库` 时，系统会检查本单内重复和库存中已存在的 IMEI/Serial。
3. 发现重复时，页面显示业务错误，库存不会生成，入库单不会关闭。
4. 操作员修正或删除重复明细后，再次点击 `确认入库`。

## 入库扫码模式与异常定位

### 变更内容

- 入库扫码页新增两种扫码模式：
  - `一版模式`：物流单号、UPC、IMEI 都填写后自动加入当前入库单。
  - `物流+UPC 模式`：物流单号和 UPC 填写后即可自动加入当前入库单，适合不逐台扫描 IMEI 的收货场景。
- 扫码输入区下方新增“上一条入库”表单，使用和当前入库相同的物流单号、UPC、IMEI 字段格式。
- “当前入库”和“上一条入库”合并在同一个面板盒子内，上一条不再作为单独提示卡展示。
- 当前入库单的“异常”统计支持点击定位：点击后会跳到第一条异常明细，并把该行数据带回上方输入区，便于手动编辑或重新扫码后重新加入。
- 异常明细行新增“修正”按钮，单条异常也可以直接带回输入区修正；原异常行仍可在草稿状态下删除。
- 后端 `POST /inbound/drafts/:id/items` 新增可选 `scanMode`：
  - `STANDARD`：默认严格模式，按商品规则要求 IMEI/Serial。
  - `TRACKING_UPC`：简化模式，物流单号和 UPC 匹配 active 商品后即可生成待确认明细。

- 前端页面：`apps/web/src/pages/inbound-scan/page.tsx`

### 使用逻辑

1. 操作员先锁定客户和仓库。
2. 默认使用 `一版模式`：扫码物流单号、UPC、IMEI 后，页面自动加入当前入库单。
3. 切换到 `物流+UPC 模式` 后，IMEI 输入框停用，只需要扫码物流单号和 UPC 即可自动加入当前入库单。
4. 页面会在扫码区下方用相同字段格式显示上一条加入的明细，便于核对刚刚扫入的数据。
5. 操作员可以点击上一条入库的“编辑”，在原位置修改物流单号、UPC、IMEI，并保存覆盖原明细。
6. 如果出现异常，点击“异常”统计或明细行“修正”，系统会定位异常行并在当前行原地编辑。
7. 操作员确认无误后再点击“确认入库”生成库存。

## 入库异常行内覆盖修正

### 变更内容

- 入库扫码页的异常“修正”改为表格行内编辑，不再把异常值带回上方扫码区重新入库。
- 行内编辑可直接修改物流单号、UPC、IMEI，点击“保存”后覆盖原 `InboundItem` 明细。
- 修正保存后复用原明细行重新校验：
  - 修正为有效商品时，原行从 `EXCEPTION` 变为 `PENDING`。
  - 修正后仍异常时，原行继续保持 `EXCEPTION`，但字段会被新值覆盖。
- 后端新增 `PATCH /inbound/drafts/:id/items/:itemId`，只允许更新 DRAFT 单里的 `PENDING` 或 `EXCEPTION` 明细。
- 覆盖修正时，原行旧的 open 异常记录会标记为 `INVALID`，再按新值重新生成当前异常记录，避免异常池里保留过期 open 异常。

## 上一条异常停止继续入库

### 变更内容

- 当前入库单最后一条有效明细如果仍是 `EXCEPTION`，扫码入库流程会暂停。
- 暂停期间，上方扫码输入、模式切换、手动加入、CSV 导入和确认入库都会被禁用。
- 页面会提示操作员必须先在下方最后一条异常行点击“修正”并保存。
- 异常统计点击时优先定位到这条阻塞异常，阻塞行会在表格中高亮。
- 后端同步加固：`POST /items`、`POST /items/import`、`POST /confirm` 都会拒绝上一条异常未处理的草稿，避免绕过前端继续入库。
- 行内修正和删除仍然允许，用于解除暂停；修正成功后继续复用原明细行，不新增数据。

### 修改位置

- 前端页面：`apps/web/src/pages/inbound-scan/page.tsx`
  - 维护阻塞异常判断、扫码/导入/确认禁用、提示文案和阻塞异常行定位。
- 前端样式：`apps/web/src/styles/globals.css`
  - 新增阻塞异常行高亮样式。
- 后端服务：`apps/api/src/modules/inbound/inbound.service.ts`
  - 新增最后一条异常阻塞校验，并在新增、导入和确认前执行。
- 后端测试：`apps/api/src/modules/inbound/tests/inbound.service.spec.ts`
  - 覆盖最后一条异常未处理时不能新增明细或确认入库。
- 产品/API/架构文档：

### 使用逻辑

3. 操作员必须在当前入库单表格中对最后一条异常点击“修正”，直接覆盖原行，或删除该异常行。
4. 修正后如果该行变为 `PENDING`，扫码、导入和确认按钮恢复可用。

## 物流单号异常确认继续入库

### 变更内容

- 当前入库的物流单号输入后会自动检查单号规则和已确认入库重复情况。
- 如果物流单号不符合 UPS/USPS/FedEx 规则，或已经存在确认入库记录，页面会提示“物流单号异常”。
- 异常提示提供“修改单号”和“继续入库”两个动作。
- 未确认前，自动加入当前单会暂停，避免异常单号直接入库。
- 点击“继续入库”后，同一个物流单号允许继续扫码 UPC/IMEI 并保存到当前入库单。
- 后端新增 `trackingExceptionConfirmed` 入参，只有操作员确认后才允许保存不符合规则的物流单号。

## 本地手机访问 API 联通修复

### 变更内容

- 本地 `.env` 的前端 API 地址从 `localhost` 调整为相对路径 `/api/v1`，解决手机访问前端后登录请求打到手机自身 `localhost` 的问题。
- Vite 本地开发服务器新增同源代理：
  - `/api` 代理到 `http://localhost:3000`
  - `/uploads` 代理到 `http://localhost:3000`
- 手机只需要访问 `http://192.168.5.239:5173/`，API、上传文件和页面都走同一个来源，避免 CORS 和手机无法访问 `3000` 端口的问题。
- 后端 CORS 开发规则补充局域网来源：
  - 继续允许 `http://localhost:5173` 和 `http://127.0.0.1:5173`。
  - 自动允许 `192.168.*.*:5173`、`10.*.*.*:5173`、`172.16-31.*.*:5173` 的本地开发来源。
- 这样电脑和手机可同时使用同一套本地 dev 服务。

### 修改位置

- 本地环境文件：`.env`
  - `WEB_ORIGIN=http://192.168.5.239:5173`
  - `VITE_API_BASE_URL=/api/v1`
- 前端 Vite 配置：`apps/web/vite.config.ts`
  - 维护本地 `/api` 和 `/uploads` 代理。
- 后端入口：`apps/api/src/main.ts`
  - 维护开发环境 CORS 来源判断。

### 使用逻辑

1. 电脑启动 `pnpm dev`。
2. 手机和电脑连接同一个 Wi-Fi。
3. 手机访问 `http://192.168.5.239:5173/`。
4. 登录请求会发送到 `http://192.168.5.239:5173/api/v1`，再由 Vite 代理转发给本地 API，不再请求手机自己的 `localhost`。
