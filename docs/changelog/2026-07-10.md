## 多人入库草稿隔离与并发保护

- `apps/api/src/modules/auth`、`apps/api/src/common/guards`、`apps/api/src/common/types`
  - 每次登录生成独立 `sessionId`，access token 和 refresh token 在同一会话内保持一致，重新登录则获得新会话。
  - JWT 守卫要求 token 必须包含 `sessionId`，并将其交给后续入库权限校验。
- `apps/api/prisma/schema.prisma`、`apps/api/prisma/migrations/20260710010000_inbound_draft_session_ownership`
  - `InboundBatch` 新增 `creatorSessionId`，建立操作人、会话、状态和更新时间组合索引。
  - 历史上没有会话字段的草稿不会批量归属：原账号第一个访问它的会话会以原子方式接管，之后其他会话不能操作。
- `apps/api/src/modules/inbound`
  - 创建、读取、恢复、扫单号、加明细、改明细、删明细、清空和确认入库都必须同时匹配操作账号与登录会话。
  - 同一个账号在多台电脑上可各自建立草稿，不再共用或互相恢复对方的草稿；不同账号同时处理同一客户时也保持各自草稿，确认后库存仍汇总到同一客户。
  - 明细变更、清空和确认会在数据库事务内锁定入库批次行；确认拿到锁后再重新读取明细并要求至少还有一条可确认货物，避免重复确认、确认同时加货、清空/删除最后一条与确认竞态，或同草稿重复 IMEI 造成错误库存。
- `apps/web/src/layouts/app-layout.tsx`、`apps/web/src/pages/inbound-scan/page.tsx`
  - 登录、退出及身份过期时清理本机的入库草稿锁定与扫码缓存，防止下一个账号看到上一个账号的本地状态。
- `apps/web/src/api/client.ts`
  - 普通业务请求在 token 刷新失败后统一返回“登录已过期，请重新登录”，同时保留后端原始错误供排查；登录/注册接口自身的认证错误仍显示原因。

### 使用逻辑

1. 每名操作员登录后创建自己的入库草稿。
2. 即使多人共用同一账号，不同登录会话也只能看到和确认自己创建的草稿。
3. 每个草稿由它的创建会话点击确认；多个草稿选择同一客户时，完成后库存在客户库存页汇总。
4. 如果同一 IMEI 被两个草稿同时录入，最终只有一个草稿能确认，另一个会收到明确的重复提示。

## 手输物流单号校验与焦点

- `apps/web/src/pages/inbound-scan/tracking-input.ts`、`tracking-input.test.ts`
  - 将单号完整性、是否可自动接受和是否需要手动确认拆成可单独测试的决策逻辑。
  - 完整 UPS 单号和 `9622` 开头的 FedEx 单号可自动进入 UPC；USPS、其他 FedEx 格式及 `BB0000` 需按 Enter 或点击继续；不完整或不符合格式时始终留在物流单号框。
- `apps/web/src/pages/inbound-scan/page.tsx`
  - 自动检查和 Enter 使用同一套规则，请求返回前还会确认输入框内容没有被用户更改。
  - 删除字符后不会再被过期的定时器抢到 UPC；异步检查期间也不会重复建草稿或重复跳转。
- `apps/api/src/modules/inbound/inbound.service.ts`
  - 扫单号响应明确返回 `formatValid` 和 `autoAccepted`，保留 `valid` 作为兼容字段。
  - 需要手动确认的有效单号仍执行重复包裹检查，不因为不自动跳转而略过后端安全校验。

### 使用逻辑

1. 手输单号时，只有达到可自动判定的完整格式才跳到 UPC。
2. 用 Backspace 删掉任一位后，焦点留在单号框，可继续修改。
3. USPS、非 `9622` FedEx 和测试单号显示手动确认按钮；明显无效的内容不显示继续按钮。

## 出库箱名稳定性与并发安全

- `apps/api/src/modules/outbound/outbound-box-name.ts`、`dto/update-outbound-box.dto.ts`
  - 箱名统一做 NFKC 和空白规范化，限制 120 个 Unicode 字符，拒绝控制字符与不可见分隔符。
  - 修改箱子必须携带 `expectedUpdatedAt`，如果其他人先保存了同一箱子，返回可读的版本冲突，不覆盖对方内容。
- `apps/api/src/modules/outbound/outbound.repository.ts`、`outbound.service.ts`
  - 建箱时在同一仓库级事务锁内完成箱号分配、规范化箱名查重、创建和审计，避免并发建箱产生相同箱号。
  - 改箱名先锁定箱子，再校验未封箱状态、版本与同仓库重名；重新打开已封箱箱子也改为原子状态更新。
- `apps/web/src/pages/outbound-packing/page.tsx`
  - 页面顶部、箱子明细弹窗和批量建箱都使用同一箱名校验，输入框限制 120 个字符。
  - 箱名输入时扫码框不再延时抢焦点；箱子明细和批量弹窗内的保存错误显示在弹窗内，不会看起来像闪退。
  - 明细弹窗保存当前箱子后，页面顶部快速编辑区会同步服务器返回的箱名、尺寸、重量和备注，避免下一次保存把旧值写回。
  - 增删箱内货物的乐观界面更新不再伪造 `updatedAt`，后续修改箱子使用真实的服务器版本。

### 使用逻辑

1. 未封箱箱子可从页面顶部或“查看明细”弹窗修改箱名。
2. 同仓库同名、输入过长或包含非法字符时，页面保留在编辑位置并显示原因。
3. 多人同时改同一箱子时，最先保存的版本成功，后续操作需刷新后重新确认。
