## Packing photo persistence recovery

### What changed

- Restored the missing `./uploads:/app/apps/api/uploads` mount in the GHCR prebuilt-image compose
  path so outbound packing photos survive API container replacement.
- Extended the required pre-deploy backup script to archive both PostgreSQL and the server
  `uploads/` directory, with the same retention window.
- Added an idempotent outbound-photo recovery script that reuses the running production images,
  backs up PostgreSQL and uploads, recreates only the API container, then verifies the bind mount,
  host/container/database counts, a real photo URL, and API health.
- Added a read-only photo audit script that compares every database photo URL with host files,
  records the missing list, checks real HTTP responses, and searches host/Docker storage for copies.
- Synchronized the production deployment documentation and mandatory agent guidance so source and
  prebuilt-image deployments follow the same photo-persistence rule.

### Touched modules and usage logic

- `docker-compose.prod.images.yml`: owns persistent API file storage for prebuilt images.
- `infra/scripts/backup-postgres.sh`: creates `wms-postgres-*.sql.gz` and
  `wms-uploads-*.tar.gz` before container replacement.
- `infra/scripts/recover-outbound-photo-storage.sh`: safely repairs a production API container that
  was started without the persistent upload mount.
- `infra/scripts/audit-outbound-photo-storage.sh`: writes a timestamped recovery report under
  `backups/` without changing photo records or files.
- `README.md`, `AGENTS.md`, and `docs/deployment/01-low-cost-vps-deployment.md`: document the
  persistent mount, backup artifacts, and recovery expectation.

This infrastructure fix prevents future photo loss. Recovery of already missing production files
must use any surviving host upload directory, old container data, server snapshots, or legacy-server
copies; database photo metadata alone does not contain the image bytes.

## Inbound note persistence, long tracking barcode, strict scan focus, and box-name printing

### What changed

- Editing a force-inbound record now loads its existing replenishment note and persists the saved
  value back to `forceReason`. The note remains visible after refresh and in customer inventory,
  while the correction continues to write an audit log.
- Box-detail printing uses the complete visible box name exactly once. Following operator feedback,
  the creation-time suffix was removed; it also does not print the customer name or a repeated short
  box label.
- Complete 34-digit numeric logistics barcodes are now supported and auto-accepted, including the
  warehouse-confirmed example `1119212621960001972000533804475274`.
- Scan focus is now sequential for every navigation path. Missing or invalid package tracking blocks
  UPC; missing or invalid UPC blocks IMEI. Enter, Tab, scanner focus changes, automatic focus, and
  mouse clicks all return the operator to the earliest incorrect field.

### Touched modules and where to modify later

- `packages/shared/src/validators/ups.ts`: shared package-tracking format and auto-accept rules.
- `apps/web/src/pages/inbound-scan/scan-fields.ts`: pure hard-stop and focus-gate decisions.
- `apps/web/src/pages/inbound-scan/page.tsx`: Enter/Tab/focus enforcement in the receiving form.
- `apps/api/src/modules/inbound/inbound.service.ts`: maps an edited forced-inbound note into the
  repository correction transaction.
- `apps/api/src/modules/inbound/inbound.repository.ts`: persists `forceReason` with the corrected
  inbound row and includes its before/after value in the audit snapshot.
- `apps/web/src/pages/inbound-records/page.tsx`: loads and labels the existing replenishment note.
- `apps/web/src/pages/outbound-packing/page.tsx`: printable heading generation from the box name.
- Related tests cover the photographed 34-digit barcode, sequential scan focus, note persistence,
  and non-repeated box-name printing.
- Product, API, and operator guidance was synchronized in `docs/product/05-inbound-rules.md`,
  `docs/product/07-outbound-rules.md`, `docs/api/08-inbound-scan.md`,
  `docs/api/11-outbound-packing.md`, and `docs/operations/user-manual.md`.

### Usage logic

1. Correct a forced-inbound row from `入库记录`; the existing replenishment note is prefilled and the
   saved text becomes the new visible `forceReason`.
2. `打印明细` begins with only the box's actual visible name, then lists product totals.
3. Scan a complete 34-digit logistics barcode normally; partial or unsupported values remain blocked.
4. Correct each scan field in order. A later field cannot receive focus while an earlier field is
   empty or invalid.
