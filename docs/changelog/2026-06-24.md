# 2026-06-24

## 入库扫码连续录入

### 变更内容

- 入库扫码页新增连续扫码焦点控制：锁定客户、切换模式、加入明细成功、修正异常成功后，光标会自动回到下一次该扫的输入框。
- 默认连续流程保持为“物流单号 -> UPC -> IMEI -> 自动加入 -> 回到物流单号”。
- `物流+UPC 模式` 保持为“物流单号 -> UPC -> 自动加入 -> 回到物流单号”。
- 新增 `同一物流单连续扫` 选项：成功加入明细后保留当前物流单号，只清空 UPC/IMEI，并把光标回到 UPC，适合同一个包裹下连续收多台机器。
- 输入框支持扫码枪常见的 Enter 结尾：物流单号回车跳 UPC，UPC 回车在一版模式跳 IMEI，在物流+UPC 模式可触发加入；IMEI 回车可触发加入。
- 如果最后一条明细是异常，或物流单号异常尚未确认，连续扫码焦点循环会暂停，避免继续录错。
- 最终 `确认入库` 仍保留为人工点击，操作员可以先核对当前入库单汇总。

### 修改位置

- 前端页面：`apps/web/src/pages/inbound-scan/page.tsx`
  - 维护扫码焦点、Enter 跳转、加入成功后的输入框清空/保留逻辑，以及同一物流单连续扫选项。
  - 后续如需调整扫码顺序、自动聚焦目标或同一物流单复用规则，优先修改这里。
- 前端样式：`apps/web/src/styles/globals.css`
  - 维护 `同一物流单连续扫` 选项的布局和输入状态。
- 产品/架构文档：
  - `docs/product/05-inbound-rules.md`
  - `docs/architecture/backend-implementation-roadmap.md`

### 使用逻辑

1. 操作员先锁定客户和仓库。
2. 正常扫码时，不需要每次用鼠标重新点击输入框；页面会按当前模式自动跳到下一步。
3. 每条明细加入成功后，页面自动准备下一条扫描。
4. 如果一个物流单下连续收多台机器，勾选 `同一物流单连续扫` 后只需要继续扫 UPC/IMEI。
5. 如果出现异常，页面暂停继续扫码，操作员先修正或删除异常行。
6. 全部明细核对无误后，操作员点击 `确认入库` 生成库存。

## 当前箱子基础信息编辑

### 变更内容

- 出库装箱页选中当前未封箱箱子后，`箱子名称` 改为可编辑输入框。
- 点击 `保存` 会更新当前箱子的箱子名称、尺寸、重量和备注，不会创建新箱子。
- 未选中箱子时，箱子名称仍显示 `创建后由系统自动生成`，点击 `新建箱子` 仍由系统生成默认箱名和内部箱号。
- 已封箱箱子保持只读，需返工重新打开后才可编辑基础信息。
- 后端 `PATCH /outbound/boxes/:id` 新增 `boxName` 入参，手动箱名不能为空，并且同一仓库内不能重复。
- 内部箱号 `boxNo` 仍始终由系统生成，改箱子名称不会改变内部追踪号。

  - 开放 `boxName` 更新字段。
- 后端服务：`apps/api/src/modules/outbound/outbound.service.ts`
  - 保存箱子名称前做空值和同仓库重名校验。
- 后端测试：`apps/api/src/modules/outbound/tests/outbound.service.spec.ts`

### 使用逻辑

1. 操作员点击 `新建箱子` 时，系统继续自动生成内部箱号和默认箱名。
2. 操作员在已创建箱子列表中选择一个未封箱箱子作为当前箱子。

## 出库细致扫码自动聚焦

### 变更内容

- 出库装箱页 `当前箱子工作区` 的 `自动扫码口` 新增连续聚焦逻辑。
- 选中或新建未封箱箱子后，页面会把光标送回 `自动扫码口`。
- 扫入第一个值后，扫码口清空并重新聚焦，UPC 或 IMEI/Serial 核对值保留在页面上。
- 只有 UPC 和 IMEI/Serial 都扫完并成功装箱后，才清空两个核对值并准备下一件货物。
- 扫码异常时保留 UPC 和 IMEI/Serial 核对值，员工点击 `清空扫码` 后再清空并回到扫码口。

### 修改位置

- 前端页面：`apps/web/src/pages/outbound-packing/page.tsx`
  - 维护细致扫码口 ref、成功扫码/成功装箱/清空扫码/切换当前箱后的回焦点逻辑。
- 产品/架构文档：
  - `docs/product/07-outbound-rules.md`
  - `docs/architecture/backend-implementation-roadmap.md`

### 使用逻辑

1. 操作员选择或新建一个未封箱箱子。
2. 光标自动进入 `自动扫码口`。
3. 扫 UPC 或 IMEI/Serial 任意一个值后，核对区显示该值，扫码口继续等待第二个值。
4. 两个值都扫完且匹配成功后，系统自动装箱并清空核对区。
5. 若匹配失败，核对区保留刚才两个值，方便员工确认错误来源。

- `pnpm --filter @wms-scan/web build`
- 本地浏览器烟测：
  - 选择本地未封箱空箱作为当前箱子后，焦点自动进入 `自动扫码口`。
  - 扫入 UPC `195950637151` 后，扫码口清空并继续聚焦，UPC 核对区保留该值，IMEI/Serial 显示等待扫描。
  - 点击 `清空扫码` 后，UPC 和 IMEI/Serial 核对区都回到等待扫描，扫码口继续聚焦。

## 装箱明细 Excel 样表排版

### 变更内容

- 明细下载中 `装箱明细` 选择 `Excel` 时，改为客户核对样表版工作簿。
- Excel 文件固定输出四个业务工作表：`出库信息`、`SN&IMEI`、`各箱型号汇总`、`出库详情`。
- `SN&IMEI` 按箱子分组列出序号、UPC、SN 和 IMEI。
- `各箱型号汇总` 按每个箱子聚合 UPC / 型号 / 数量并显示箱内合计。
- `出库信息` 和 `出库详情` 显示客户、出库日期、总数量、箱数和整份导出的 UPC / 型号汇总。
- CSV 导出保持原有字段勾选逻辑；只有 `OUTBOUND_DETAIL + EXCEL` 使用固定样表排版。

### 修改位置

- 后端服务：`apps/api/src/modules/reports/reports.service.ts`
  - 维护普通 CSV / Excel XML 导出，以及装箱明细 Excel 的四工作表生成逻辑。
  - 后续如需调整客户样表列名、sheet 顺序、颜色、列宽或合计规则，优先修改这里。
- 后端测试：`apps/api/src/modules/reports/tests/reports.service.spec.ts`
  - 覆盖装箱明细 Excel 的 sheet 名称、按箱分组、SN/IMEI 明细和总数输出。
- 产品/API/架构文档：
  - `docs/product/10-report-rules.md`
  - `docs/api/14-reports.md`
  - `docs/architecture/backend-implementation-roadmap.md`

### 使用逻辑

1. 操作员进入 `明细下载`。
2. 报表类型选择 `装箱明细`，格式选择 `Excel`。
3. 根据需要选择客户、时间范围、封箱状态或搜索条件。
4. 点击预览确认范围，再点击生成导出。
5. 下载的 Excel 可直接用于客户核对：先看出库信息，再看每箱 SN/IMEI，最后核对各箱和整票 UPC/型号汇总。

## 作废箱子名称释放

### 变更内容

- 出库箱子名称重复校验改为只检查同仓库内未作废箱子。
- 已作废箱子的可见箱子名称会释放出来，新箱子或当前未封箱箱子可以继续使用这个名称。
- 箱名重复错误提示改为中文：`当前仓库已存在同名箱子，请修改箱子名称后再保存。`

### 修改位置

- 后端仓库层：`apps/api/src/modules/outbound/outbound.repository.ts`
  - `findBoxByName` 排除 `VOIDED` 箱子。
- 后端服务：`apps/api/src/modules/outbound/outbound.service.ts`
  - 维护箱名重复的中文业务提示。
- 后端测试：`apps/api/src/modules/outbound/tests/outbound.service.spec.ts`
  - 覆盖重复箱名中文提示，以及作废箱名可复用。
- 产品/API 文档：
  - `docs/product/07-outbound-rules.md`
  - `docs/api/11-outbound-packing.md`

### 使用逻辑

1. 操作员删除或作废一个箱子后，这个箱子的可见箱名不再占用。
2. 后续新建箱子或编辑未封箱箱子时，可以重新使用该名称。
3. 如果同仓库仍有未作废箱子正在使用该名称，系统会用中文提示阻止保存。
