## Module Purpose And Usage Logic

### apps/api

Backend API application. It is organized by module:

- `config`: environment and application configuration.
- `common`: reusable backend infrastructure such as guards, filters, interceptors, errors, pipes, logger, and utilities.
- `database`: Prisma service and database access boundary.
- `health`: health check endpoint.
- `jobs`: future queue and background processors.
- `modules`: business modules such as auth, customers, products, inbound, inventory, outbound, exceptions, reports, audit logs, and settings.

Future backend changes should go into the matching business module. Controllers handle HTTP, services handle business rules, repositories handle persistence.

### apps/web

Frontend application skeleton. It is organized by:

- `app`: providers and router.
- `api`: API client and query client.
- `layouts`: application layouts.
- `pages`: route-level pages.
- `features`: business feature logic.
- `components`: reusable UI and workflow components.
- `styles`: global CSS and design tokens.

Future UI work should add page-level code under `pages` and reusable business logic under `features`.

### packages/shared

Shared TypeScript package for frontend and backend contracts:

- Business enums
- API response types
- Scan code types
- Scan validation helpers
- Business constants

Use this package when frontend and backend need the same enum, type, or validation rule.

## How To Run Later

Dependencies have been declared but not installed in this change.

After approval, install dependencies with:

```bash
pnpm install
```

Local infrastructure can be started with:

```bash
docker compose up -d
```

Once dependencies are installed, use:

```bash
pnpm dev
pnpm lint
pnpm typecheck
pnpm test
```
