## Module Purpose

- `apps/api/src/modules/warehouses`: owns warehouse profile management, active flags, timezone metadata, and warehouse change audit records.
- `apps/api/src/modules/settings`: owns system-level configuration for default warehouse, inbound scan lock behavior, outbound customer ownership enforcement, duplicate detection, exception creation, notifications, and retention days.
- `apps/api/src/modules/settings/settings.keys.ts`: owns stable database setting keys and defaults that future inbound, outbound, exception, report, and cleanup jobs should reuse.
- `apps/api/src/modules/customers`: owns customer profile management, active/inactive status, reusable selector data, customer summary counts, and customer change audit records.
- `apps/api/src/modules/products`: owns Apple product master data, SKU identity, UPC mappings, UPC lookup for inbound recognition, product activation state, import behavior, and UPC product change audit records.
- `apps/api/prisma/seed.ts`: owns development baseline records for permissions, admin user, default warehouse, demo customer, demo UPC, and system settings.
- `docs/api/05-warehouses-settings.md`: documents phase-four request and response contracts.
- `docs/api/06-customers.md`: documents phase-five customer request and response contracts.
- `docs/api/07-products-upc-library.md`: documents phase-six product library request and response contracts.
- `docs/product/02-system-settings-rules.md`: documents product rules for system settings and their downstream business meaning.

- `docs/product/04-upc-product-rules.md`: documents SKU, UPC, active/inactive, IMEI flag, import, deletion, and audit rules.

## Usage Logic

After local PostgreSQL is migrated and seeded, log in with the seeded development admin and call protected phase-four endpoints with the returned bearer token.

Warehouse settings page can call:

```text
GET /api/v1/warehouses
POST /api/v1/warehouses
PATCH /api/v1/warehouses/:id
GET /api/v1/settings
PATCH /api/v1/settings
```

Example grouped settings update:

```json
{
  "scanRules": {
    "requiresLockedCustomer": true,
    "enforceOutboundCustomerOwnership": true,
    "detectDuplicateImei": true,
    "detectDuplicateUps": true
  },
  "retention": {
    "auditLogRetentionDays": 365,
    "reportExportRetentionDays": 30,
    "exceptionRecordRetentionDays": 730
  }
}
```

The settings API stores stable key-value rows in `system_settings`, but returns grouped data for frontend ergonomics.

Customer management and downstream customer selectors can call:

```text
GET /api/v1/customers
GET /api/v1/customers/options
GET /api/v1/customers/:id
POST /api/v1/customers
PATCH /api/v1/customers/:id
PATCH /api/v1/customers/:id/status
GET /api/v1/customers/:id/summary
```

Inactive customers are preserved for historical records. New inbound selectors should call `GET /api/v1/customers/options` without `includeInactive` so inactive customers are excluded by default.

UPC product library and downstream UPC recognition can call:

```text
GET /api/v1/products
GET /api/v1/products/:id
GET /api/v1/products/by-upc/:upc
POST /api/v1/products
PATCH /api/v1/products/:id
PATCH /api/v1/products/:id/status
POST /api/v1/products/import
```

`GET /api/v1/products/by-upc/:upc` returns only active product and active UPC mappings, so phase-seven inbound scanning can treat a not-found response as an unmatched UPC exception candidate.
