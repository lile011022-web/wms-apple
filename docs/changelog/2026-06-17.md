## Usage Logic

After local PostgreSQL is migrated and seeded, log in with the seeded development admin and call protected inbound endpoints with the returned bearer token.

Inbound scan pages can call:

```text
POST /api/v1/inbound/drafts
GET /api/v1/inbound/drafts/:id
POST /api/v1/inbound/drafts/:id/ups
POST /api/v1/inbound/drafts/:id/items
DELETE /api/v1/inbound/drafts/:id/items/:itemId
DELETE /api/v1/inbound/drafts/:id/items
POST /api/v1/inbound/drafts/:id/confirm
```

Inbound record, batch customer-change, and detail-download pages can call:

```text
GET /api/v1/inbound/records
GET /api/v1/inbound/records/:id
GET /api/v1/inbound/records/:id/items
POST /api/v1/inbound/records/export-preview
```

Example record filters:

```json
{
  "customerId": "customer-1",
  "warehouseId": "warehouse-1",
  "status": "CONFIRMED",
  "inventoryStatus": "IN_STOCK",
  "upsTrackingNo": "1Z999AA10123456784",
  "upc": "194253149189",
  "imei": "356789012345678",
  "dateFrom": "2026-06-01T00:00:00.000Z",
  "dateTo": "2026-06-30T23:59:59.999Z"
}
```

`GET /inbound/records` returns normalized row data, linked inventory status, exception summaries, and `selectableForCustomerChange`.

`POST /inbound/records/export-preview` returns the estimated row count and a reusable report payload so the future reports module can create export jobs with the same filter semantics.

`POST /inbound/drafts/:id/confirm` rechecks duplicates inside a database transaction. Only confirmable rows create `inventory_items`; duplicate rows are marked `EXCEPTION`; the batch is marked `CONFIRMED`; and an `INBOUND_CONFIRM` audit log is written.
