## Usage Logic

After local PostgreSQL is migrated and seeded, log in with the seeded development admin and call protected inbound endpoints with the returned bearer token.

Inbound scan pages can call:

```text
POST /api/v1/inbound/drafts
GET /api/v1/inbound/drafts/:id
POST /api/v1/inbound/drafts/:id/ups
POST /api/v1/inbound/drafts/:id/items
DELETE /api/v1/inbound/drafts/:id/items/:itemId
DELETE /api/v1/inbound/drafts/:id/items
POST /api/v1/inbound/drafts/:id/confirm
```

Inbound record, batch customer-change, and detail-download pages can call:

```text
GET /api/v1/inbound/records
GET /api/v1/inbound/records/:id
GET /api/v1/inbound/records/:id/items
POST /api/v1/inbound/records/export-preview
```

Example record filters:

```json
{
  "customerId": "customer-1",
  "warehouseId": "warehouse-1",
  "status": "CONFIRMED",
  "inventoryStatus": "IN_STOCK",
  "upsTrackingNo": "1Z999AA10123456784",
  "upc": "194253149189",
  "imei": "356789012345678",
  "dateFrom": "2026-06-01T00:00:00.000Z",
  "dateTo": "2026-06-30T23:59:59.999Z"
}
```

`GET /inbound/records` returns normalized row data, linked inventory status, exception summaries, and `selectableForCustomerChange`.

`POST /inbound/records/export-preview` returns the estimated row count and a reusable report payload so the future reports module can create export jobs with the same filter semantics.

`POST /inbound/drafts/:id/confirm` rechecks duplicates inside a database transaction. Only confirmable rows create `inventory_items`; duplicate rows are marked `EXCEPTION`; the batch is marked `CONFIRMED`; and an `INBOUND_CONFIRM` audit log is written.

Customer inventory, outbound packing, detail-download, and dashboard pages can call:

```text
GET /api/v1/inventory/customer-summary
GET /api/v1/inventory/products
GET /api/v1/inventory/products/:productId/items
GET /api/v1/inventory/items
GET /api/v1/inventory/items/:id
GET /api/v1/inventory/available-for-outbound
GET /api/v1/inventory/export-preview
```

Example inventory filters:

```json
{
  "customerId": "customer-1",
  "warehouseId": "warehouse-1",
  "status": "IN_STOCK",
  "upc": "194253149189",
  "imei": "356789012345678",
  "search": "iPhone 16 Pro"
}
```

`GET /inventory/customer-summary` returns customer status totals and SKU count. `GET /inventory/products` returns SKU/product-level counts. `GET /inventory/items` returns item-level IMEI/Serial rows. `GET /inventory/available-for-outbound` requires `customerId` and forces `IN_STOCK`, so exception, packed, outbound, and voided rows are not selectable for outbound packing.

Outbound packing pages can call:

```text
POST /api/v1/outbound/boxes
GET /api/v1/outbound/boxes
GET /api/v1/outbound/boxes/:id
GET /api/v1/outbound/available-items
POST /api/v1/outbound/boxes/:id/items
DELETE /api/v1/outbound/boxes/:id/items/:itemId
DELETE /api/v1/outbound/boxes/:id/items
POST /api/v1/outbound/boxes/:id/seal
```

Example outbound box creation:

```json
{
  "customerId": "customer-1",
  "warehouseId": "warehouse-1",
  "boxNo": "BOX-20260617-001",
  "notes": "Packing lane A"
}
```

`GET /outbound/available-items` requires `customerId` and reuses inventory availability logic. `POST /outbound/boxes/:id/items` only accepts `IN_STOCK` inventory from the same customer and warehouse as the open box, then marks it `PACKED`. Removing or clearing an open box returns affected inventory to `IN_STOCK`. `POST /outbound/boxes/:id/seal` requires at least one item, seals inside a database transaction, and writes an `OUTBOUND_BOX_SEAL` audit log.

Exception pool and dashboard pages can call:

```text
GET /api/v1/exceptions
GET /api/v1/exceptions/summary
GET /api/v1/exceptions/:id
POST /api/v1/exceptions/:id/resolve
POST /api/v1/exceptions/:id/ignore
POST /api/v1/exceptions/:id/invalidate
POST /api/v1/exceptions/batch-resolve
POST /api/v1/exceptions/batch-ignore
```

Example exception filters:

```json
{
  "status": "OPEN",
  "type": "UPC_NOT_MATCHED",
  "customerId": "customer-1",
  "warehouseId": "warehouse-1",
  "search": "884909876543"
}
```

Resolve, ignore, and invalidate actions require `resolutionNote`. Only `OPEN` exceptions can be handled. Batch endpoints process each requested ID independently and return one result per ID, so a non-open or missing exception does not block other valid records. Successful handling writes an `EXCEPTION_HANDLE` audit log.

Batch customer-change pages can call:

```text
GET /api/v1/customer-changes/candidates
POST /api/v1/customer-changes/preview
POST /api/v1/customer-changes/commit
GET /api/v1/customer-changes/logs
```

Example preview payload:

```json
{
  "currentCustomerId": "customer-1",
  "newCustomerId": "customer-2",
  "inboundItemIds": ["item-1", "item-2"]
}
```

`POST /customer-changes/preview` returns `previewToken`, affected rows, blocked rows, and impact counts. `POST /customer-changes/commit` requires the same selected rows, a required reason, and the preview token. Commit rechecks the selected rows, rejects stale previews, blocks packed/outbound inventory, updates inbound/inventory/exception customer ownership in one transaction, and writes `CustomerChangeLog` plus `CUSTOMER_BATCH_CHANGE` audit entries.

Detail-download and report-export pages can call:

```text
POST /api/v1/reports/preview
POST /api/v1/reports/exports
GET /api/v1/reports/exports
GET /api/v1/reports/exports/:id
GET /api/v1/reports/exports/:id/download
```

Example inventory export payload:

```json
{
  "reportType": "INVENTORY_DETAIL",
  "format": "CSV",
  "filters": {
    "customerId": "customer-1",
    "warehouseId": "warehouse-1",
    "inventoryStatus": "IN_STOCK"
  },
  "fields": ["customerCode", "warehouseCode", "sku", "imei", "serial", "status"]
}
```

`POST /reports/preview` returns estimated row count, selected fields, and the allowed field list for the report type. `POST /reports/exports` creates a completed CSV or Excel XML export for small result sets, rejects unsupported fields, blocks result sets over the synchronous row limit for future background jobs, stores generated file metadata in `ReportExport`, and writes a `REPORT_EXPORT` audit log. `GET /reports/exports` is scoped to the requester. `GET /reports/exports/:id/download` only allows the requester to download completed exports.
