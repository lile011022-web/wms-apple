# 2026-06-30

## Inbound warehouse compensation tracking

### What changed

- Added a `BB0000` warehouse compensation package tracking rule for inbound scanning.
- Operators can manually enter package numbers that start with `BB0000` when the warehouse
  compensates a customer with a replacement package.
- The scan page now treats `BB0000` compensation package numbers as auto-accepted logistics values,
  normalizes them to uppercase, and allows the row to proceed without the unsupported-format warning.
- Backend inbound validation accepts `BB0000` package numbers without requiring
  `trackingExceptionConfirmed`, while keeping normal duplicate tracking checks in place.

### Touched modules

- `packages/shared/src/validators/ups.ts`: shared package tracking validation and auto-accept rules.
- `packages/shared/src/validators/scan.test.ts`: validator coverage for `BB0000` compensation
  tracking normalization, validity, and auto-accept behavior.
- `apps/api/src/modules/inbound/tests/inbound.service.spec.ts`: inbound service coverage for
  `BB0000` scan checks and item saving.
- `apps/web/src/pages/inbound-scan/page.tsx`: inbound tracking placeholder, completion detection,
  and abnormal-format warning wording.
- `docs/product/05-inbound-rules.md`: product rule for warehouse compensation package tracking.
- `docs/api/08-inbound-scan.md`: API contract for `/inbound/drafts/:id/ups` and add-item behavior.

### Usage logic

1. In 入库扫码, manually enter a package number starting with `bb0000`, optionally followed by an
   internal customer or package suffix.
2. The system stores the value as uppercase, for example `bb0000jh05` becomes `BB0000JH05`.
3. The row can continue like a normal accepted package tracking number.
4. If the same compensation number already exists in confirmed records or the current draft, the
   normal duplicate logistics warning still applies.

## Cross-customer inventory ownership visibility

### What changed

- Added an all-customer lookup mode to the customer inventory page so operators can search unknown
  packages without clicking every customer one by one.
- Changed inventory SKU summaries to group by customer plus SKU instead of only SKU, preventing the
  same product from different customers from being merged in all-customer views.
- Added visible customer ownership columns to customer inventory SKU rows, customer inventory detail
  rows, outbound packing available-inventory rows, and exception-pool details.
- Expanded inventory search to include customer code and customer name, alongside tracking number,
  inbound batch, box number/name, UPC, IMEI, Serial, SKU, and product name.
- Kept outbound packing and destructive inventory cleanup bound to one selected customer; all-customer
  views are for lookup and ownership identification, not mixed-customer packing or deletion.

### Touched modules

- `apps/api/src/modules/inventory/inventory.repository.ts`: inventory grouping and broad search
  filters, including customer code/name.
- `apps/api/src/modules/inventory/inventory.service.ts`: all-customer summary support and
  customer-aware SKU summary response mapping.
- `apps/api/src/modules/inventory/tests/inventory.service.spec.ts`: coverage for all-customer
  summary, customer-aware SKU rows, and customer-name search filters.
- `apps/web/src/pages/customer-inventory/page.tsx`: all-customer selector option, customer columns,
  and guarded cleanup controls.
- `apps/web/src/pages/outbound-packing/page.tsx`: customer column and customer-aware search prompt in
  the available-inventory table.
- `apps/web/src/pages/exception-pool/page.tsx`: customer code/name display in the exception list and
  handling panel.
- `docs/product/06-inventory-rules.md`, `docs/product/07-outbound-rules.md`, `docs/api/10-inventory.md`,
  and `docs/api/11-outbound-packing.md`: product and API rules for cross-customer lookup.

### Usage logic

1. In 客户库存, choose `全部客户` to search unknown packages across customers.
2. Use the customer column to identify ownership before deciding whether to switch to a specific
   customer for cleanup or downstream operations.
3. In 出库装箱, keep selecting the target customer for actual packing, while the visible customer
   column makes inventory rows and screenshots self-identifying.
4. In 异常池, use `全部客户` plus the customer column to locate unknown exception packages without
   manually cycling through customers.

## Outbound all-customer lookup

### What changed

- Added a real `全部客户` option to the 出库装箱 customer selector.
- In `全部客户` mode, the available-inventory table shows paginated inventory across customers for
  the selected warehouse, and the search box narrows that global list.
- Global outbound lookup rows show customer ownership, tracking number, IMEI/Serial, product,
  status, and latest box context, so unknown packages or exception-like rows can be located without
  opening customers one by one.
- Disabled selection, new box creation, scan packing, batch packing, and add-to-box actions while
  `全部客户` is selected. Operators must switch to the row's specific customer before packing.

### Touched modules

- `apps/web/src/pages/outbound-packing/page.tsx`: customer selector all-customer mode, global
  search branch, read-only result table behavior, and packing action guards.
- `docs/product/07-outbound-rules.md`: product rule for global lookup versus customer-scoped packing.
- `docs/api/11-outbound-packing.md`: API routing note that all-customer lookup uses
  `GET /inventory/items`, while packing remains on customer-scoped outbound endpoints.

### Usage logic

1. In 出库装箱, choose `全部客户`.
2. Review the paginated all-customer rows, or enter a UPS/USPS/FedEx number, IMEI/Serial, customer
   name, UPC, SKU, or product keyword to narrow the list.
3. Use the row's `客户`, `状态`, and `所在箱` context to locate the package.
4. If the row needs actual packing, switch the top customer selector to that customer first, then
   create/select a box and continue normal装箱.

## Created-box workspace filtering

### What changed

- Added created-box view filters to 出库装箱: `未完成`, `仓库今日`, `近7仓库日`, `已封箱`, and
  `全部历史`.
- The default created-box view is now `未完成`, which sends `status = OPEN` and prevents sealed
  history from piling up in the active packing workspace.
- `仓库今日` and `近7仓库日` calculate UTC date ranges from the selected warehouse timezone, so China
  users and US warehouse users see the same warehouse business-date grouping.
- The created-box list cache now respects the active view; sealing a box removes it from the
  default unfinished list, while reopening can bring it back.

### Touched modules

- `apps/api/src/modules/outbound/dto/list-outbound-boxes-query.dto.ts`: added `createdFrom` and
  `createdTo` query filters.
- `apps/api/src/modules/outbound/outbound.service.ts`: outbound box list filtering by created date
  range.
- `apps/api/src/modules/outbound/tests/outbound.service.spec.ts`: coverage for status and created
  date range filters.
- `apps/web/src/pages/outbound-packing/page.tsx`: created-box view tabs, warehouse-timezone date
  range calculation, and view-aware cache updates.
- `apps/web/src/styles/globals.css`: created-box view tab styling.
- `docs/product/07-outbound-rules.md` and `docs/api/11-outbound-packing.md`: product and API rules
  for created-box filtering.

### Usage logic

1. Enter 出库装箱; the created-box list starts on `未完成`.
2. Use `仓库今日` to see boxes created on the selected warehouse's current business date.
3. Use `近7仓库日` for recent work without depending on China/browser time.
4. Use `已封箱` or `全部历史` only when searching or downloading historical boxes.
