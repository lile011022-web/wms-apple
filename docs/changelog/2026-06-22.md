# 2026-06-22 Changelog

## Faster production deployment path

### What changed

- Added step timing to the production deploy script so slow deploys show whether time is spent on Git update, image build or pull, Prisma migration, or container restart.
- Added selective service deployment: `infra/scripts/deploy.sh web` for frontend-only releases and `infra/scripts/deploy.sh api` for backend releases.
- Kept full deployment as the default when no service is passed.
- Added `RUN_MIGRATIONS=auto|always|never`; auto runs migrations for full and API deployments, but skips them for web-only deployments.
- Enabled Docker BuildKit by default during deploy and added pnpm store cache mounts to the API and web Dockerfiles.
- Added an optional GHCR prebuilt-image deployment path with `docker-compose.prod.images.yml`, `USE_PREBUILT_IMAGES=true`, `WEB_IMAGE`, and `API_IMAGE`.
- Added deployment support for externally built API and web images published to GitHub Container Registry.

### Where to modify later

- `infra/scripts/deploy.sh`: production deploy timing, selected service build/pull/start behavior, migration policy, and prebuilt-image switch.
- `apps/api/Dockerfile` and `apps/web/Dockerfile`: production image build layers and pnpm cache behavior.
- `docker-compose.prod.images.yml`: prebuilt-image compose file used when the VPS should pull images instead of building locally.
- `docs/deployment/01-low-cost-vps-deployment.md`, `README.md`, and `AGENTS.md`: operator-facing deploy commands and safety notes.

### Module purpose

- The deploy script owns repeatable VPS releases and should keep production secrets, database backups, migrations, container restarts, and health-check expectations explicit.
- The Dockerfiles own production image construction for the NestJS API and nginx-served Vite frontend.
- The prebuilt-image path moves expensive image building away from the small VPS while preserving the same runtime compose topology.

### Usage logic

1. For normal releases, sync the checkout to `/opt/wms-scan`, back up PostgreSQL, then run `PROJECT_DIR=/opt/wms-scan infra/scripts/deploy.sh`.
2. For frontend-only changes, run `PROJECT_DIR=/opt/wms-scan infra/scripts/deploy.sh web`; migrations are skipped automatically.
3. For backend or Prisma changes, run `PROJECT_DIR=/opt/wms-scan infra/scripts/deploy.sh api`; migrations run automatically.
4. For prebuilt-image releases, publish API and web images externally, configure `WEB_IMAGE` and `API_IMAGE` in server-only `.env.production`, then run `USE_PREBUILT_IMAGES=true COMPOSE_FILE=docker-compose.prod.images.yml PROJECT_DIR=/opt/wms-scan infra/scripts/deploy.sh`.
5. Always verify containers and `http://24.199.87.181/api/v1/health` after deployment.

## Detail-download time filtering and batch-named inbound downloads

### What changed

- Added start-time and end-time filters to the 明细下载 page.
- Detail-download previews and exports now pass `dateFrom` and `dateTo` to the existing report API filters.
- The page blocks preview/export when the end time is earlier than the start time, and requires a fresh preview after filter changes.
- Added an inbound batch selector to the 明细下载 page when the report type is 入库明细.
- Added a report API batch option endpoint so the selector lists confirmed inbound batches by customer and recent confirmation time.
- Inbound detail exports filtered by batch now include the batch number in the downloaded file name.

### Where to modify later

- `apps/web/src/pages/detail-download/page.tsx`: report filter UI, time range conversion, batch selector behavior, and browser download handling.
- `apps/api/src/modules/reports/reports.controller.ts`: report routes, including inbound batch options.
- `apps/api/src/modules/reports/reports.service.ts`: export preview, export creation, file naming, and response shaping.
- `apps/api/src/modules/reports/reports.repository.ts`: report database filters and inbound batch option queries.
- `docs/api/14-reports.md` and `docs/product/10-report-rules.md`: API contract and report workflow rules.

### Module purpose

- The web detail-download page lets operators preview, generate, and re-download business detail files.
- The reports backend owns export permissions, field whitelists, reusable filters, time-range filtering, file content, file names, and audit logs.
- The inbound batch option query exposes confirmed batch choices without requiring operators to guess IDs.

### Usage logic

1. Open 明细下载 and choose 入库明细.
2. Optionally choose a customer, start time, and end time to narrow the report rows.
3. Choose 全部批次 or a specific 入库批次.
4. Preview the row count and sample data after changing filters.
5. Generate the export; selected-batch files are named like `inbound_detail-INB-20260622-001-export-id.csv`.

## Outbound packing photo before sealing

### What changed

- Added packing photo upload and delete support to 已创建箱子 on the 出库装箱 page.
- Split photo capture into 相册 and 拍照 controls so mobile operators can either select an existing image or open the rear camera directly.
- Added mobile-photo upload hardening: large phone photos are compressed to JPEG before upload, and the photo upload request has a longer timeout than normal API calls.
- Hardened production static asset caching so mobile browsers do not keep stale SPA HTML after deploys, and missing old JS/CSS assets return `404` instead of `index.html`.
- Switched the web production bundle to stable `assets/index.js` and `assets/index.css` names, with Nginx fallbacks for stale mobile asset URLs after deploys.
- Added a visible HTML loading fallback and React error boundary so mobile browser runtime failures show an error message instead of a blank white screen.
- Fixed outbound packing photo URL rendering when production uses the relative API base `/api/v1`, preventing uploaded-photo cards from crashing mobile browsers with `"/api/v1" cannot be parsed as a URL`.
- Added a production `/uploads/` Nginx proxy route and API upload persistence mount so uploaded outbound box photos render from the web origin and survive API container replacement.
- Changed mobile photo actions to native `label`-driven file inputs instead of programmatically clicking hidden inputs, improving iOS Safari and embedded browser behavior.
- Restored default warehouse behavior on inbound scan, customer inventory, and outbound packing by reading `warehouse.defaultId` from system settings before falling back to the first active warehouse.
- Adjusted created-box photo controls so photo count, upload buttons, delete, and shipment-number actions wrap inside the card instead of overflowing.
- Added an 上传单号 control beside photo upload so operators can save the outbound shipment or label number on the box.
- Added packing photo and shipment number visibility inside 查看明细 so operators can review evidence from the detail modal.
- Added `outbound_box_photos` storage metadata and audit actions for photo add/delete.
- Blocked outbound box sealing unless the open box contains at least one item and at least one packing photo.
- Exposed uploaded photos in outbound box list/detail responses so operators can see photo evidence before sealing.

### Where to modify later

- `apps/web/src/pages/outbound-packing/page.tsx`: created-box photo UI, mobile camera/gallery upload controls, delete controls, and seal-before-photo prompt.
- `apps/web/src/api/workflow.ts`: outbound photo upload preparation, mobile photo compression, multipart form submission, and upload timeout.
- `apps/web/src/api/client.ts`: shared request options, including per-request timeout overrides.
- `nginx/default.conf`: production SPA cache headers and static asset fallback behavior.
- `docker-compose.prod.yml`: production API upload directory persistence.
- `.gitignore`: uploaded evidence files remain server/runtime data and are not committed.
- `apps/web/vite.config.ts`: stable production asset file names to prevent stale mobile HTML from referencing missing hashed bundles.
- `apps/web/index.html`, `apps/web/src/main.tsx`, and `apps/web/src/app/error-boundary.tsx`: visible loading and render-error fallbacks for mobile blank-screen diagnosis.
- `apps/web/src/styles/globals.css`: mobile-friendly file input labels and created-box photo action layout.
- `apps/web/src/utils/default-warehouse.ts`: shared default warehouse selection helper.
- `apps/web/src/pages/inbound-scan/page.tsx`, `apps/web/src/pages/customer-inventory/page.tsx`, and `apps/web/src/pages/outbound-packing/page.tsx`: default warehouse selection from system settings.
- `apps/web/src/pages/outbound-packing/page.tsx`: created-box shipment number editor beside photo upload and detail-modal evidence review.
- `apps/api/src/modules/outbound/outbound.controller.ts`: outbound photo upload/delete routes.
- `apps/api/src/modules/outbound/outbound.service.ts`: photo validation, file storage, and seal prerequisite rule.

### Module purpose

- The outbound packing page lets warehouse operators create boxes, pack inventory, upload box-content evidence, seal boxes, and reopen sealed boxes for rework.
- The outbound backend owns customer/warehouse ownership checks, inventory state changes, photo evidence requirements, and audit logs.
- The upload directory stores image files while the database keeps searchable metadata linked to each outbound box.

### Usage logic

1. Create or open an outbound box.
2. Add the customer inventory rows to the box.
3. In 已创建箱子, use 相册 to upload an existing photo or 拍照 on mobile to capture a new box-content photo. Large phone photos are compressed before upload so slow mobile networks do not fail as easily.
4. Click 上传单号 beside the photo controls and save the outbound shipment or label number.
5. Use 查看明细 to review the saved photos and single outbound shipment number.
6. Confirm sealing only after at least one photo is present.
7. If the box is reopened for rework, update the photo evidence and shipment number before sealing again.

## FedEx package validation and force inbound

### What changed

- Documented FedEx package tracking support for common 12, 15, 20, and 22 digit numeric formats plus longer 96-prefixed SmartPost/Ground Economy style numbers.
- Added a controlled `POST /inbound/records/:id/force-confirm` endpoint for exception inbound rows.
- Added force-inbound fields on inbound items: `forcedInbound`, `forceReason`, `forcedAt`, and `forcedById`.
- Added an `INBOUND_FORCE_CONFIRM` audit action.
- Added an inbound records page action that lets operators force-confirm eligible exception rows after entering a reason.

### Where to modify later

- `packages/shared/src/validators/ups.ts`: package tracking validation rules for UPS, USPS, and FedEx.
- `apps/api/src/modules/inbound/inbound.service.ts`: force-inbound business checks.
- `apps/api/src/modules/inbound/inbound.repository.ts`: force-inbound inventory creation, exception resolution, and audit transaction.
- `apps/web/src/pages/inbound-records/page.tsx`: inbound records force-inbound action and operator prompt.
- `docs/api/08-inbound-scan.md`, `docs/product/05-inbound-rules.md`, and `docs/database/05-inbound-inventory-transaction.md`: API, product, and database behavior.

### Module purpose

- The shared package-tracking validator keeps carrier number rules reusable across API and UI workflows.
- The inbound backend owns whether exception rows may become inventory and preserves auditability.
- The inbound records page gives supervisors a visible recovery path without changing normal draft confirmation logic.

### Usage logic

1. Normal inbound still requires UPC/product match and valid IMEI or Serial before confirmation.
2. FedEx numeric tracking values with 12, 15, 20, 22, or longer 96-prefixed SmartPost/Ground Economy style digits pass package tracking validation.
3. If a confirmed batch leaves a matched row in `EXCEPTION`, open 入库记录 and click 强制入库.
4. Enter the manual review reason.
5. The system rechecks duplicate IMEI or Serial, creates inventory, resolves the row's open exceptions, and writes an `INBOUND_FORCE_CONFIRM` audit log.
