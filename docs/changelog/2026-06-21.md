# 2026-06-21

## Outbound Packing Horizontal Workbench

### What Changed

- Rebuilt `/outbound-packing` as a horizontal outbound packing workbench.
- Replaced the previous vertical form flow with a compact first-row header, a one-line current box quick editor, two large side-by-side operation panels, and a horizontal created-box card list.
- Kept one unified tracking search field for UPS, FedEx, and USPS package numbers.
- Added the requested box size presets with `12 × 12 × 12 in` as the default.
- Kept weight defaulted to `45 lb`.
- Added `手动调整`; length, width, and height inputs are only shown after that control is opened.
- Added a customer inventory table with search, 10 / 20 / 50 page size, pagination, carrier display, item status badges, and `加入箱子`.
- Added a current box workspace with summary tiles, search, pagination, delete controls, detail view entry, and reorder action.
- Replaced inline history details with a detail drawer/modal opened from the box icon or `查看明细`.
- Added horizontal created-box cards with status-specific actions for 未封箱, 已封箱, and 返工中.
- Added CSS-only animations for card entry, box icon click, row highlight after adding, and row fade during removal.

### Touched Modules

- `apps/web/src/pages/outbound-packing/page.tsx`: owns the new workbench layout, component split, existing outbound API integration, local rework-state mapping, search, pagination, modal, and packing interactions.
- `apps/web/src/styles/globals.css`: adds scoped `outbound-` workbench styles, table layout, status badges, card list, modal, and lightweight animations.

### Usage Logic

1. Open `/outbound-packing`.
2. Select the customer in the top header and search UPS, FedEx, or USPS tracking text from the unified search field.
3. Use the compact quick editor to prepare the current box, keeping the default `12 × 12 × 12 in` and `45 lb` unless changed.
4. Click `手动调整` only when a custom length, width, and height are needed.
5. Add available inventory from the left customer inventory panel to the current box workspace on the right.
6. Delete items only while the box is 未封箱 or 返工中.
7. Confirm sealing from the quick editor, current box card, or rework flow.
8. Open box details from the box icon or `查看明细`; sealed boxes are read-only, while editable boxes allow deletion from the modal.

### Backend Coordination Notes

- The current API still stores tracking number under the legacy `upsTrackingNo` field; the page treats it as a unified package tracking field for UPS, FedEx, and USPS.
- The backend status enum currently returns `OPEN` and `SEALED`; the frontend maps reopened sealed boxes to local `返工中` state after `POST /outbound/boxes/:id/reopen`.
- The backend supports only `12*12*12`, `14*14*14`, and `CUSTOM` size values today; the frontend maps other preset labels to `CUSTOM` with a generated custom size string.

- Each product summary shows total quantity, IMEI / Serial quantity, and UPS / FedEx / USPS counts.

## Outbound Box Name Uniqueness And UPC Summary

### What Changed

- Box names are now treated as unique inside the same warehouse.
- The API rejects duplicate non-empty box names when creating or updating outbound boxes.
- The outbound workbench checks duplicate box names before submitting so operators get immediate feedback.
- The current box workspace now supports selecting multiple packed rows and running one batch delete action.
- The old bottom single-item delete action was replaced by batch delete to avoid repeating the row-level delete button.
- The box detail summary now groups strictly by UPC, not by product name plus UPC.
- Each UPC summary shows packed quantity in the box, currently available quantity left in customer inventory, IMEI / Serial quantity, and UPS / FedEx / USPS counts.

### Touched Modules

- `apps/api/src/modules/outbound`: enforces warehouse-level outbound box name uniqueness in create and update flows.
- `apps/api/src/modules/outbound/tests/outbound.service.spec.ts`: covers duplicate box name rejection.
- `apps/web/src/pages/outbound-packing/page.tsx`: adds frontend duplicate-name validation, current-box row selection, batch delete, and UPC-only detail summaries.
- `apps/web/src/styles/globals.css`: adds checkbox column sizing for the current box workspace.

### Usage Logic

1. Enter a box name before 新建箱子 or 暂存.
2. If another box in the same warehouse already uses that name, the operation is blocked.
3. In 当前箱子工作区, select one or more rows and click 批量删除.
4. Open 查看明细 to review UPC-level counts and remaining available inventory by UPC.

## Created Box Selection And Confirmed Delete

### What Changed

- Created box cards can now be selected by clicking the card.
- Added a `删除选中` action to the 已创建箱子 toolbar.
- Deleting selected boxes opens a second confirmation modal before any API call is made.
- Deleting a box is implemented as a safe void operation instead of a physical database delete.
- Open or rework boxes return their packed inventory to available stock when deleted.
- Sealed boxes are blocked from direct deletion and must be reopened first.
- Box deletion writes an `OUTBOUND_BOX_DELETE` audit log.

### Touched Modules

- `apps/api/prisma/schema.prisma` and `apps/api/prisma/migrations/20260620000000_outbound_box_edit_rework/migration.sql`: add the delete audit action.
- `apps/api/src/modules/outbound`: exposes `DELETE /outbound/boxes/:id`, voids open boxes transactionally, returns packed inventory to stock, and blocks sealed-box deletion.
- `apps/web/src/api/workflow.ts`: exposes `outboundApi.deleteBox`.
- `apps/web/src/pages/outbound-packing/page.tsx`: adds created-box selection state, delete confirmation modal, and selected-box delete flow.
- `apps/web/src/styles/globals.css`: styles selected box cards, toolbar delete action, and the confirmation modal.

### Usage Logic

1. Click one or more cards in 已创建箱子 to select them.
2. Click 删除选中.
3. Review the second confirmation modal.
4. Click 确认删除 to void the selected boxes and return any open-box inventory to stock.
5. If a selected box is 已封箱, reopen it first before deleting.

## Production API Image Slimming

### What Changed

- Changed the API production Dockerfile to build a focused `pnpm deploy` package instead of copying the entire monorepo builder workspace into the runtime image.
- Kept Prisma CLI available in the deployed API package so production migration commands can still run from the API image.
- Reduced server-side image export pressure during VPS deployment.
- Fixed the production API image package step so the generated `.prisma/client` directory is copied into the focused runtime package after `pnpm deploy`.

### Touched Modules

- `apps/api/Dockerfile`: creates `/prod/api` during the builder stage, copies the generated Prisma Client into that deployed package, and copies only the focused package into the runner stage.

### Usage Logic

1. Build the API image with `docker compose -p wms-scan -f docker-compose.prod.yml --env-file .env.production build api`.
2. Run production database migrations from the rebuilt API image before replacing the running API container.
3. Start the stack normally with `infra/scripts/deploy.sh`.

## Customer Inventory Batch Packing And Sealed Detail Export

### What Changed

- Added batch packing controls to `/customer-inventory`.
- Customer inventory now supports selecting warehouse-scoped available detail rows, creating or selecting an open outbound box, batch adding selected rows to that box, and sealing the box.
- The customer inventory page still reuses the outbound APIs for box creation, item packing, and sealing, so ownership, warehouse, duplicate, status, and audit checks remain centralized.
- Changed the detail-download label from 出库明细 to 装箱明细.
- Detail download now offers `仅已封箱` for 装箱明细 and sends `outboundStatus = SEALED` so downloads can target sealed box contents only.
- Outbound report search now also matches UPC and package tracking number.
- Outbound detail exports now include box name, customer name, product name, and package tracking number fields.

### Touched Modules

- `apps/web/src/pages/customer-inventory/page.tsx`: owns customer inventory summary/detail display plus the new warehouse selection, open-box selection, batch packing, and sealing shortcut.
- `apps/web/src/pages/detail-download/page.tsx`: owns report type selection, sealed-only outbound detail filter, preview, export creation, and browser download trigger.
- `apps/api/src/modules/reports/reports.repository.ts`: owns report row filtering and now includes UPC/tracking search for outbound detail reports.
- `apps/api/src/modules/reports/reports.service.ts`: owns report field whitelists and export formatting; outbound detail fields now include box/customer/product/tracking context.
- `docs/api/10-inventory.md` and `docs/product/06-inventory-rules.md`: document customer inventory batch packing rules.
- `docs/api/14-reports.md` and `docs/product/10-report-rules.md`: document sealed outbound detail export behavior.

### Usage Logic

1. Open `/customer-inventory`.
2. Select customer and warehouse.
3. In `未封箱箱子`, choose an open box or click `新建箱子`.
4. Select rows in `IMEI 明细`; only rows marked 可出库 can be checked.
5. Click `批量装箱` to add the selected rows to the chosen open box.
6. Click `封箱` when the box is ready.
7. Open `/detail-download`, choose `装箱明细`, keep `仅已封箱`, preview, then generate CSV or Excel to download sealed box contents.
