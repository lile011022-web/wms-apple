# 2026-07-02

## Inbound duplicate error hardening

### What changed

- Added a global Prisma database-error mapping so unique-constraint conflicts return a stable API
  error envelope instead of `Internal server error`.
- Hardened inbound draft confirmation so inventory IMEI/Serial uniqueness conflicts caught during
  the final transaction are converted into operator-readable duplicate messages.
- Blocked inbound-record corrections while the source batch is still a draft, preventing a single
  row correction from partially confirming an unfinished draft batch.
- Added draft recovery by visible inbound batch number so operators can restore an older open
  `INB-...` draft from the scan page when the browser is pointing at a newer empty draft.
- Fixed the inbound package-tracking DTO so the exact warehouse compensation value `BB0000` is not
  rejected by generic minimum-length validation before the business rule runs.
- Extended the inbound confirmation transaction window for large restored batches and mapped Prisma
  transaction timeout errors to an operator-readable response.
- Documented that confirmation must keep duplicate identity failures actionable even when the
  database constraint is the final guard.

### Touched modules

- `apps/api/src/common/filters/all-exceptions.filter.ts`: converts Prisma `P2002` and `P2025`
  errors into API contract responses, and maps transaction timeouts to a large-batch retry message.
- `apps/api/src/common/filters/all-exceptions.filter.spec.ts`: verifies the `P2002` conflict
  envelope and transaction-timeout response.
- `apps/api/src/modules/inbound/inbound.repository.ts`: translates transaction-time IMEI/Serial
  uniqueness conflicts during inventory creation into the same Chinese correction guidance used by
  the precheck path, gives large confirmation transactions enough time to complete, and refuses
  record corrections against draft batches.
- `apps/api/src/modules/inbound/inbound.service.ts`: rejects inbound-record corrections for draft
  batches before repository writes and restores open drafts by batch number.
- `apps/api/src/modules/inbound/inbound.controller.ts`: exposes
  `GET /inbound/drafts/by-batch/:batchNo`.
- `apps/api/src/modules/inbound/tests/inbound.service.spec.ts`: covers the draft-batch correction
  rejection, batch-number draft restore behavior, and exact `BB0000` package tracking.
- `apps/api/src/modules/inbound/dto/scan-inbound-ups.dto.ts`: allows 6-character compensation
  package numbers at request-validation level.
- `packages/shared/src/validators/scan.test.ts`: covers exact `BB0000` normalization and
  auto-acceptance.
- `apps/web/src/api/workflow.ts`: adds the frontend draft restore request helper.
- `apps/web/src/pages/inbound-scan/page.tsx`: adds the operator-facing restore control in the
  current draft panel.
- `apps/web/src/styles/globals.css`: styles the restore control so it remains visible beside the
  active draft number.
- `docs/product/05-inbound-rules.md`: product rule for readable duplicate failures during
  confirmation.
- `docs/api/08-inbound-scan.md`: API contract note for duplicate identity conflict responses.

### Usage logic

1. Operators still fix duplicate IMEI/Serial rows by editing or deleting the preview row before
   confirming.
2. If precheck catches the duplicate, confirmation returns the existing readable duplicate message.
3. If the database catches the duplicate during final inventory creation, confirmation now returns
   the same actionable duplicate message instead of a generic server error.
4. If a draft row needs correction before confirmation, operators use 入库扫码 edit/delete controls;
   入库记录 correction is only for rows from already confirmed batches.
5. If the page shows a newer empty draft, paste the old `INB-...` number into `按入库单号恢复` and
   click `恢复草稿` to switch the page back to that open batch.
6. Operators may enter `BB0000` by itself for warehouse compensation packages; it now passes both
   request validation and the shared package-tracking rule.
7. Large restored drafts can be confirmed in one operation without hitting Prisma's short default
   interactive-transaction timeout; if the database still times out, the API explains the batch-size
   retry path instead of showing a generic database failure.

## Customer inventory detail cleanup

### What changed

- Moved operator-facing customer inventory deletion from the SKU summary area to the item-level
  customer inventory detail table.
- Added `DELETE /inventory/items` so operators delete exact IMEI/Serial inventory rows by
  `inventory_items.id`.
- Kept product catalog data safe: detail deletion does not delete UPCs, SKUs, products, customers,
  inbound batches, or inbound item history.
- Blocked deletion of packed/outbound inventory detail rows so shipping history cannot be broken
  from the customer inventory page.

### Touched modules

- `apps/api/src/modules/inventory/dto/delete-inventory-items.dto.ts`: request contract for selected
  inventory detail IDs.
- `apps/api/src/modules/inventory/inventory.controller.ts`: exposes `DELETE /inventory/items` under
  `customers.manage`.
- `apps/api/src/modules/inventory/inventory.service.ts`: normalizes customer, warehouse, and item ID
  inputs before deletion.
- `apps/api/src/modules/inventory/inventory.repository.ts`: validates ownership/status, clears
  inbound/exception inventory pointers, deletes selected inventory rows, and writes an audit log.
- `apps/api/src/modules/inventory/tests/inventory.service.spec.ts`: covers selected detail-row
  deletion and empty selection rejection.
- `apps/web/src/api/workflow.ts`: adds the frontend `deleteItems` API helper.
- `apps/web/src/pages/customer-inventory/page.tsx`: removes destructive controls from the SKU
  summary table and adds selection/delete controls to the customer inventory detail table.
- `docs/product/06-inventory-rules.md`: defines item-level deletion as the customer inventory page
  cleanup boundary.
- `docs/api/10-inventory.md`: documents `DELETE /inventory/items` and clarifies that SKU deletion is
  not the preferred operator workflow.

### Usage logic

1. Operators select a specific customer before destructive cleanup.
2. In 客户库存明细, operators tick exact IMEI/Serial rows, then use `删除选中` or `删除当前分页`.
3. The confirmation prompt states that UPC/SKU/product data will not be deleted.
4. The backend rejects rows that are already packed, outbound, or linked to outbound box items.
5. Successful deletion refreshes customer totals, SKU summary counts, and detail rows.

## WMS PDF optimization pass

### What changed

- Changed the customer inventory locator from a single business date to start/end datetime inputs so
  operators can narrow customer inventory to an exact time window.
- Kept the customer inventory detail shortcut export aligned with the visible table and included
  `入库时间` in the downloaded fields.
- Added an inbound Excel layout for `飞书入库登记表`, exporting fixed headers `入库时间`, `单号`,
  `UPC`, `IMEI`, `商品名称`, and `数量`.
- Changed the inventory Excel hold layout to `未封箱留仓汇总表格`: it now exports `PACKED`
  inventory whose related outbound box is still `OPEN`, matching the "已装箱未封箱" review need.
- Added outbound box type/size filtering to the detail-download box selector and returns
  `sizePreset/customSize` in box options.
- Updated the default inventory-detail report header from `单号, UPC, IMEI, 商品名称, 数量` to
  `单号, 入库时间, UPC, IMEI, 商品名称, 数量`.

### Touched modules

- `apps/web/src/pages/customer-inventory/page.tsx`: owns customer inventory time filtering,
  status-card drilldown, visible detail rows, and the `导出明细` shortcut.
- `apps/web/src/pages/detail-download/page.tsx`: owns report type selection, Excel table modes,
  outbound box picking, and box type/size filtering.
- `apps/api/src/modules/reports/reports.service.ts`: owns report layout selection, export file
  generation, inventory summary rows, and the new inbound registration worksheet.
- `apps/api/src/modules/reports/reports.repository.ts`: owns report row and outbound box option
  queries, including inventory filtering by related outbound box status.
- `apps/api/src/modules/reports/dto/*`: documents accepted report layout and box option query
  values.
- `apps/api/src/modules/reports/tests/reports.service.spec.ts`: locks the new report headers and
  Excel layouts.
- `docs/product/06-inventory-rules.md`: records customer inventory time filtering and export
  expectations.
- `docs/product/10-report-rules.md`: records report layout behavior and the unsealed hold rule.
- `docs/api/14-reports.md`: records API fields, layout values, and outbound box option filters.

### Usage logic

1. In 客户库存, operators choose start/end time, click a summary card such as 在库 or 已装箱, and
   use `导出明细` to download rows matching the current visible filters.
2. In 明细下载, choose `入库明细` + `Excel` + `飞书入库登记表` for the six-column customer/Feishu
   registration shape.
3. In 明细下载, choose `库存明细` + `Excel` + `未封箱留仓汇总表格` to review items already added to
   boxes but not yet sealed.
4. In 明细下载 > 装箱明细, use customer/time/search/status plus `箱子类型` to narrow the candidate
   boxes before selecting exact boxes for preview and export.
