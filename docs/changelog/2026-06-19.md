# 2026-06-19

## UPS/USPS/FedEx Inbound Package Tracking

### What Changed

- Expanded inbound package tracking validation from UPS-only to UPS, USPS, and FedEx tracking numbers.
- Kept the existing `upsTrackingNo` API/database field for backward compatibility, while documenting its business meaning as a package tracking number.
- Updated the inbound scan page to label the field as `物流单号` with `UPS / USPS / FedEx` guidance and to show the tracking number in the current draft preview table.
- Updated inbound API and product documentation so operators and future developers do not treat this flow as UPS-only.

### Touched Modules

- `packages/shared/src/validators/ups.ts`: owns tracking-number format validation and normalization shared by frontend/backend packages.
- `apps/api/src/modules/inbound/inbound.service.ts`: normalizes inbound package tracking values, rejects unsupported formats, and passes normalized values into duplicate checks and draft items.
- `apps/api/src/modules/inbound/dto`: documents the compatible `upsTrackingNo` request field and allows longer USPS/FedEx formats.
- `apps/web/src/pages/inbound-scan/page.tsx`: owns the operator-facing inbound scan form and draft preview.
- `docs/api/08-inbound-scan.md` and `docs/product/05-inbound-rules.md`: describe the API contract and product rule for package tracking.

### Usage Logic

1. The operator locks a customer and warehouse before scanning.
2. The operator can enter or scan UPS, USPS, or FedEx tracking into the logistics field.
3. The backend removes spaces, uppercases letters, validates the value, and stores it in the compatible `upsTrackingNo` field.
4. Duplicate package tracking checks continue to use the same confirmed inbound record lookup, so existing exception and reporting flows remain compatible.

### Later Maintenance Notes

- If the project later needs carrier-specific reporting, add an explicit carrier field instead of overloading `upsTrackingNo`.
- If USPS/FedEx barcode subtypes need stricter checksum validation, extend `packages/shared/src/validators/ups.ts` and keep `parseBarcode` stable so UPC/IMEI are not misclassified as package tracking.
