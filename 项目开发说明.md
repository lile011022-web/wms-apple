# WMS Scan 项目开发说明

## 1. 项目定位

本项目是面向美国仓库 Apple 产品的扫码入库、库存管理、出库装箱与异常处理系统。当前仓库内已有 `前端页面/` 静态高保真原型，共 11 个页面，覆盖 Dashboard、入库扫码、入库记录、客户库存、出库装箱、异常池、批量修改客户、明细下载、UPC 商品库、客户管理和系统设置。

正式开发目标是将现有 HTML 原型升级为可登录、可扫码、可追踪、可审计、可导出报表的 Web WMS 系统。

## 2. 当前原型分析

### 2.1 核心页面

| 模块 | 原型页面 | 业务作用 |
| --- | --- | --- |
| 运营概览 | `index.html` | 展示今日入库、库存、异常、趋势和操作日志 |
| 入库扫码 | `inbound-scan.html` | 客户锁定后扫描 UPS、UPC、IMEI，生成待确认入库明细 |
| 入库记录 | `inbound-records.html` | 查询历史入库数据，支持筛选、导出、批量操作 |
| 客户库存 | `customer-inventory.html` | 按客户查看 SKU 汇总、IMEI 明细和异常数量 |
| 出库装箱 | `outbound-packing.html` | 按客户选择库存明细加入箱号并封箱 |
| 异常池 | `exception-pool.html` | 处理 UPC 未匹配、IMEI 重复、UPS 重复、客户归属错误、IMEI 未入库 |
| 批量修改客户 | `batch-customer-change.html` | 批量修正入库记录客户归属并记录日志 |
| 明细下载 | `detail-download.html` | 按条件导出入库、出库、库存报表 |
| UPC 商品库 | `upc-library.html` | 管理 UPC 与 Apple 商品信息映射 |
| 客户管理 | `customer-management.html` | 管理客户资料、状态和库存入口 |
| 系统设置 | `system-settings.html` | 仓库、用户权限、扫码规则、数据保留 |

### 2.2 关键业务规则

1. 入库必须先选择并锁定客户，后续 UPS、UPC、IMEI 都绑定该客户。
2. UPC 必须能映射到商品库，否则进入异常池。
3. 需要 IMEI 的商品必须录入唯一 IMEI。
4. 系统需要自动检测 IMEI 重复和 UPS 重复。
5. 出库装箱时，IMEI 必须属于当前选择客户，且状态为在库。
6. 批量修改客户必须保留影响范围和修改日志。
7. 所有关键操作需要可审计，包括操作员、时间、前后值和原因。

## 3. 推荐整体架构

建议采用前后端分离架构：

```text
浏览器 / 扫码枪
  -> 前端管理后台
  -> 后端 API 服务
  -> PostgreSQL 数据库
  -> Redis 缓存与任务锁
  -> 文件存储 / 报表导出
```

### 3.1 前端技术栈

推荐：

- React 19 + TypeScript：适合后台系统组件化开发，生态成熟。
- Vite：开发启动快，构建配置简单。
- React Router：管理 11 个业务页面路由。
- TanStack Query：管理接口请求、缓存、刷新和错误状态。
- Zustand：管理扫码会话、客户锁定、临时入库预览等轻量状态。
- Ant Design 或 Arco Design：提供表格、表单、弹窗、上传、分页、权限菜单等后台组件。
- Tailwind CSS 或 CSS Modules：若继续沿用当前视觉设计，可将 `design-system.css` 拆成设计变量和组件样式。
- Zod：前端表单和扫码输入校验，与后端 DTO 规则保持一致。
- Playwright：端到端测试关键业务流。
- Vitest + React Testing Library：组件和工具函数测试。

不建议第一版使用 Next.js，除非后续需要服务端渲染或复杂门户页面。当前是仓库内网/后台系统，Vite SPA 更轻、更直接。

### 3.2 后端技术栈

推荐：

- Node.js 22 LTS + NestJS + TypeScript：前后端同语言，模块化强，适合权限、审计、事务、队列等中后台系统。
- Prisma ORM：类型安全，迁移清晰，适合快速定义客户、商品、库存、出入库等关系模型。
- PostgreSQL 16+：库存、流水、审计类数据需要强一致事务和唯一约束。
- Redis：扫码会话缓存、短期幂等键、导出任务状态、限流。
- BullMQ：异步报表导出、批量导入 UPC、批量客户变更。
- JWT + Refresh Token 或企业 SSO：第一版可用账号密码登录，后续可扩展 Google Workspace / Microsoft Entra ID。
- OpenAPI / Swagger：自动生成接口文档，方便前后端协作。
- Pino：结构化日志。
- ExcelJS：导出 Excel 报表。
- Multer 或 S3 SDK：处理 UPC 批量导入、异常图片、包裹照片等文件。

如果团队更熟悉 Java，也可以采用 Spring Boot 3 + PostgreSQL + Redis。考虑到前端是 TypeScript，且项目需要快速迭代，首选 NestJS。

## 4. 后端模块设计

### 4.1 模块划分

| 模块 | 职责 |
| --- | --- |
| Auth | 登录、刷新令牌、退出、密码策略 |
| Users & Roles | 用户、角色、权限、仓库绑定 |
| Warehouses | 仓库基础信息、时区、扫码规则 |
| Customers | 客户管理、客户状态、客户库存入口 |
| Products | UPC 商品库、分类、是否需要 IMEI |
| Inbound | 入库扫码会话、入库确认、入库记录 |
| Inventory | 在库库存、IMEI 明细、SKU 汇总 |
| Outbound | 箱号创建、装箱、封箱、出库记录 |
| Exceptions | 异常创建、处理、忽略、人工关联商品 |
| Batch Change | 批量修改客户、影响预览、审批与日志 |
| Reports | 入库、出库、库存明细导出 |
| Audit Logs | 关键操作审计日志 |
| System Settings | 扫码规则、保留策略、通知配置 |

### 4.2 核心数据表

第一版建议至少包含以下表：

- `users`：用户账号、姓名、邮箱、状态、最后登录时间。
- `roles`、`permissions`、`role_permissions`：角色权限模型。
- `warehouses`：仓库编号、名称、地址、时区。
- `customers`：客户编号、名称、联系人、状态。
- `products`：商品名称、品牌、型号、分类、颜色、容量、是否需要 IMEI。
- `product_upcs`：UPC、商品 ID、状态、更新时间。
- `inbound_batches`：一次入库确认批次，绑定客户、仓库、操作员。
- `inbound_items`：UPS、UPC、IMEI、商品、客户、入库状态。
- `inventory_items`：每个可库存单元，通常以 IMEI/序列号为核心。
- `outbound_boxes`：箱号、客户、仓库、状态、创建人、封箱时间。
- `outbound_items`：箱号内明细，绑定库存项。
- `exceptions`：异常类型、原始扫码内容、关联 UPS/UPC/IMEI、状态、处理结果。
- `customer_change_logs`：批量改客户前后值、原因、影响数量、操作员。
- `audit_logs`：操作类型、资源类型、资源 ID、前后快照、IP、User-Agent。
- `export_jobs`：导出任务状态、筛选条件、文件地址、过期时间。

### 4.3 数据约束

必须放在数据库层兜底：

- `product_upcs.upc` 唯一。
- 已入库且未作废的 `inventory_items.imei` 唯一。
- 同一仓库下有效 `outbound_boxes.box_no` 唯一。
- 出库明细不能重复引用同一个未出库库存项。
- 所有库存状态变更必须在数据库事务内完成。

## 5. 关键业务流程

### 5.1 入库扫码

1. 操作员选择客户并锁定。
2. 扫描 UPS，系统校验格式和是否重复。
3. 扫描 UPC，系统查询商品库。
4. 如果 UPC 未匹配，创建异常记录，前端提示人工处理。
5. 扫描 IMEI，系统校验格式、唯一性、是否已入库。
6. 前端将本次扫码明细放入临时预览区。
7. 点击确认入库后，后端开启事务：
   - 创建入库批次。
   - 写入入库明细。
   - 创建或更新库存项。
   - 创建审计日志。
8. 返回确认结果和异常列表。

### 5.2 出库装箱

1. 选择客户。
2. 创建或选择箱号。
3. 查询该客户可出库库存。
4. 勾选 UPS 包裹或具体 IMEI。
5. 后端校验库存状态、客户归属和重复装箱。
6. 封箱确认后，库存项状态改为已出库或已装箱，写入出库流水。

### 5.3 异常处理

异常类型建议枚举：

- `UPC_NOT_FOUND`
- `IMEI_DUPLICATED`
- `UPS_DUPLICATED`
- `CUSTOMER_MISMATCH`
- `IMEI_NOT_IN_STOCK`

处理动作建议枚举：

- `RESOLVE`
- `IGNORE`
- `MARK_INVALID`
- `LINK_PRODUCT`
- `CHANGE_CUSTOMER`

每次处理必须记录处理人、处理时间、处理备注和处理前后数据。

## 6. API 设计建议

### 6.1 接口风格

使用 REST API，路径统一使用 `/api/v1` 前缀。

示例：

```text
POST   /api/v1/auth/login
GET    /api/v1/customers
POST   /api/v1/products/upcs/import
GET    /api/v1/products/upcs
POST   /api/v1/inbound/scan/validate
POST   /api/v1/inbound/batches
GET    /api/v1/inbound/records
GET    /api/v1/inventory
GET    /api/v1/inventory/items
POST   /api/v1/outbound/boxes
POST   /api/v1/outbound/boxes/:boxId/items
POST   /api/v1/outbound/boxes/:boxId/seal
GET    /api/v1/exceptions
POST   /api/v1/exceptions/:id/resolve
POST   /api/v1/reports/export
GET    /api/v1/audit-logs
```

### 6.2 返回格式

统一返回结构：

```json
{
  "success": true,
  "data": {},
  "requestId": "req_20260617_001"
}
```

错误返回：

```json
{
  "success": false,
  "error": {
    "code": "IMEI_DUPLICATED",
    "message": "IMEI 已存在，不能重复入库",
    "details": {}
  },
  "requestId": "req_20260617_002"
}
```

## 9. 代码风格规范

### 9.1 通用规范

- 全项目使用 TypeScript 严格模式。
- 文件名使用 kebab-case，例如 `inbound-scan.service.ts`。
- 类型、类、React 组件使用 PascalCase。
- 变量、函数使用 camelCase。
- 枚举值使用大写蛇形，例如 `IMEI_DUPLICATED`。
- 不在业务代码中硬编码状态文案，统一使用枚举和映射。
- 所有时间存储 UTC，前端按仓库时区展示。
- 所有金额、数量、库存状态变更都不能只依赖前端判断。

### 9.2 前端规范

- 页面只负责布局和组合，业务逻辑下沉到 `features/`。
- API 请求统一走 `api/client.ts`，自动附带 token、requestId 和错误处理。
- 表单校验使用 Zod schema，避免散落的 if 判断。
- 表格筛选条件同步到 URL query，方便刷新和分享。
- 扫码输入组件独立封装，统一处理自动聚焦、回车提交、节流和错误提示。
- 不在组件里直接拼接后端错误文案，用错误码映射为用户可读文本。

### 9.3 后端规范

- Controller 只处理路由、DTO 和权限，不写业务逻辑。
- Service 负责业务规则，Repository 或 Prisma 层负责数据访问。
- 所有写入类接口都要记录审计日志。
- 所有库存状态变更必须使用事务。
- 所有批量操作必须支持 dry-run 或 preview，先展示影响范围。
- 所有导入导出任务异步执行，避免阻塞 API 请求。

## 10. 安全性要求

### 10.1 身份与权限

- 后台系统必须登录后访问。
- 使用 RBAC 权限控制，不同角色只能访问对应菜单和接口。
- 关键权限建议包括：入库确认、出库封箱、异常处理、批量改客户、用户管理、系统设置、报表导出。
- 登录失败需要限流，防止暴力破解。
- Refresh Token 需要可撤销，退出登录后立即失效。

### 10.2 数据安全

- 后端必须校验所有输入，不能相信前端。
- IMEI、UPS、UPC 等扫码内容需要长度、字符集和业务规则校验。
- 数据库唯一约束必须兜底防重复。
- 导出文件需要权限校验和过期时间。
- 异常图片、包裹照片等上传文件需要限制类型、大小和访问权限。
- 所有敏感配置放在环境变量或密钥管理系统中，不提交到 Git。

### 10.3 操作审计

以下操作必须写入审计日志：

- 登录、退出、登录失败。
- 确认入库。
- 封箱出库。
- 异常处理。
- 批量修改客户。
- UPC 商品库新增、修改、停用。
- 用户、角色、权限变更。
- 系统设置变更。
- 报表导出。

## 11. 可维护性设计

### 11.1 可扩展

- 将仓库、客户、商品、库存、出入库、异常拆成独立模块。
- 状态机集中管理库存状态，避免页面里各自判断。
- 异常类型和处理动作使用枚举，后续新增异常不影响老数据。
- 导出任务独立为队列，后续可接入 S3、邮件通知或定时任务。

### 11.2 可测试

推荐测试分层：

- 单元测试：扫码规则、状态流转、DTO 校验、权限判断。
- 集成测试：入库事务、出库事务、异常处理事务。
- E2E 测试：登录、入库扫码、出库装箱、异常处理、导出报表。
- 数据库迁移测试：确保迁移能从空库和已有库稳定执行。

### 11.3 可观测

- 每个请求生成 `requestId`。
- 日志记录用户 ID、仓库 ID、客户 ID、资源 ID、耗时和错误码。
- 关键指标包括：入库确认耗时、出库封箱耗时、异常数量、重复扫码次数、导出任务耗时。
- 生产环境接入 Sentry 或 OpenTelemetry。

## 13. 本项目建议的第一步

建议下一步先创建正式工程骨架：

```text
apps/
  web/       # React 前端
  api/       # NestJS 后端
packages/
  shared/    # 前后端共享类型、枚举、Zod schema
docs/
  api/
  database/
  product/
```

第一版先不要急着重写所有页面，可以按优先级迁移：

1. 登录与布局。
2. UPC 商品库。
3. 客户管理。
4. 入库扫码。
5. 客户库存。
6. 出库装箱。
7. 异常池。

这样可以先把基础数据打通，再做扫码主流程，风险最低。
